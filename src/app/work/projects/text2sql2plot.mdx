## Overview

Text2SQL2Plot is an LLM-powered analytics agent designed for turbomachinery maintenance teams in the oil & gas industry. It enables engineers to query complex time-series data using natural language, automatically generating SQL queries and visualizations to accelerate diagnostics and reduce repair time.

## Problem Statement

Turbomachinery engineers needed to analyze vast amounts of sensor data to diagnose issues, but lacked SQL expertise. Traditional BI tools required extensive training and couldn't handle the domain-specific queries engineers needed. This created bottlenecks in the diagnostic process, delaying repairs and increasing downtime costs.

## Solution

An intelligent agent that:
1. Accepts natural language queries (e.g., "Show me vibration trends for Compressor A over the last 30 days")
2. Translates queries into optimized SQL using LLM with domain-specific context
3. Executes queries against time-series database
4. Generates appropriate visualizations (line charts, heatmaps, scatter plots)
5. Provides diagnostic insights based on patterns

## Key Features

- **Domain-Aware Translation**: Fine-tuned LLM with turbomachinery domain knowledge for accurate SQL generation
- **Visualization Intelligence**: Automatically selects appropriate chart types based on query intent and data characteristics
- **Error Handling**: Gracefully handles ambiguous queries with clarifying questions
- **Performance Optimization**: Query optimization layer ensures fast execution on large time-series datasets
- **Audit Trail**: Logs all queries and results for compliance and analysis

## Technologies Used

- **Python**: Core implementation language
- **LangChain**: LLM orchestration and prompt engineering
- **OpenAI GPT-4**: Natural language understanding and SQL generation
- **TimescaleDB**: Time-series database for sensor data
- **Plotly**: Interactive visualization generation
- **FastAPI**: REST API for integration with existing tools
- **Docker**: Containerized deployment

## Technical Challenges

1. **Domain Adaptation**: Generic LLMs struggled with turbomachinery terminology. Solution: Created domain-specific few-shot examples and fine-tuned prompts.
2. **Query Ambiguity**: Natural language is inherently ambiguous. Solution: Implemented clarification dialog system and confidence scoring.
3. **Performance at Scale**: Large time-series queries were slow. Solution: Added intelligent query optimization and result caching.
4. **Reliability**: Production systems require high accuracy. Solution: Implemented validation layer and fallback mechanisms.

## Architecture

```
User Query → LLM (Intent + SQL Generation) → Validator → TimescaleDB
                                                              ↓
User ← Visualization ← Chart Generator ← Result Processor ← Query Result
```

## Lessons Learned

- **Domain expertise is critical**: Generic LLMs need domain-specific context to be production-ready
- **User trust requires transparency**: Showing generated SQL builds confidence
- **Error handling is paramount**: Graceful degradation is more important than perfect accuracy
- **Iteration with users**: Continuous feedback loops dramatically improved query accuracy
